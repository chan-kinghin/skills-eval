### Why Case 2 Is Hard: Surcharge Label Precision / 为什么附加费检测难：标签精度

**The Problem / 问题:**
The expected output requires `"label": "Mirror coating"`. Models add extra context.

**What models actually returned / 模型实际返回内容:**

| Model | Output 输出 | Error 错误 |
|---|---|---|
| qwen-turbo (0%) | `"Mirror coating (M suffix)"` | Added suffix explanation / 添加了后缀说明 |
| qwen-max (67%) | Trial 3: `"Mirror coating (M suffix)"` | Inconsistent across trials / 跨试验不一致 |
| qwen-plus (67%) | Trial 2: `"mirror"` | Lowercase, incomplete / 小写且不完整 |
| MiniMax-M2 (33%) | `"Mirror coating (M suffix)"` | Same extra context / 同样多余上下文 |
| MiniMax-M2.5 (33%) | `"Mirror coating (M suffix) +$0.45"` | Added price too / 还添加了价格 |
| **MiniMax-Text-01 (100%)** | `"Mirror coating"` | **Exact match / 精确匹配** |

**Root Cause / 根本原因:**

This is NOT a conceptual failure — every model correctly identified that 8120M needs a mirror surcharge. The failure is in **output format conformance**. Models treat the `label` field as an opportunity to explain their reasoning, adding parenthetical context like `(M suffix)` or the dollar amount `+$0.45`.

这**不是**概念性失败——每个模型都正确识别了 8120M 需要电镀附加费。失败在于**输出格式一致性**。模型把 `label` 字段当作解释推理的机会，添加了括号说明如 `(M suffix)` 或金额 `+$0.45`。

**Why this matters for skills / 对技能的启示:**

The skill prompt defines surcharge labels as `"镜片电镀 Mirror coating"` in the rules table. When models see a definition like `Mirror coating: M suffix → +$0.45`, they conflate the label, the trigger condition, and the amount into a single string. This reveals a **prompt design issue**: the skill doesn't explicitly say "use only the label text, nothing else."

技能提示在规则表中定义了附加费标签为`"镜片电镀 Mirror coating"`。当模型看到类似 `Mirror coating: M suffix → +$0.45` 的定义时，它们会将标签、触发条件和金额合并为一个字符串。这揭示了一个**提示设计问题**：技能没有明确说"仅使用标签文本，不要附加其他内容"。

**Fix / 修复建议:**

```
Surcharge rules (use ONLY the label text in the "label" field, no amounts or trigger explanations):
附加费规则（"label" 字段仅使用标签文本，不要添加金额或触发条件说明）：
```

### Why Case 4 Is Hard: Count vs. Sum Ambiguity / 为什么多附加费计数难：计数与求和歧义

**The Problem / 问题:**
Expected: `"total_surcharges": 5` (count of instances).
Models returned: `2.88` or `4.83` (sum of dollar amounts).

**What models actually returned / 模型实际返回内容:**

| Model | Trial 1 | Trial 2 | Trial 3 | Interpretation 解读 |
|---|:---:|:---:|:---:|---|
| MiniMax-M2.5 (0%) | `2.88` | `2.88` | `2.33` | Sum of surcharge amounts / 附加费金额之和 |
| MiniMax-Text-01 (0%) | `4.83` | `4.03` | `4.83` | Partial sums / 部分金额求和 |

**The arithmetic check / 算术验证:**
- 5 surcharges: mirror ($0.45) + cartoon ring ($0.33) + water transfer ($0.50) + spray ($1.15) + chrome ($0.45)
- Sum = $2.88 — exactly what MiniMax-M2.5 returned ✓

The model's arithmetic is **correct** — it just answered a different question.

模型的算术是**正确的**——它只是回答了另一个问题。

**Root Cause / 根本原因:**

The field name `total_surcharges` is semantically ambiguous. In financial contexts, "total" almost always means "sum of monetary values." But the test expects a count. The prompt says "Total surcharges detected: at least 4 individual surcharge applications" — the word "individual" suggests counting, but "total" suggests summing.

字段名 `total_surcharges` 在语义上有歧义。在财务语境中，"total（总计）"几乎总是意味着"金额之和"。但测试期望的是计数。提示语说"Total surcharges detected: at least 4 individual surcharge applications"——"individual"暗示计数，但"total"暗示求和。

**Why MiniMax fails more than Qwen / 为什么 MiniMax 比通义千问更容易失败:**

MiniMax models appear to have stronger financial domain training, causing them to default to monetary aggregation. Qwen models (especially turbo) treat the field more literally as a JSON counter.

MiniMax 模型似乎有更强的金融领域训练，导致它们默认进行金额聚合。通义千问模型（尤其是 turbo）更字面地将该字段视为 JSON 计数器。

**Fix / 修复建议:**

Use unambiguous field names:
使用无歧义的字段名：

```json
// Bad 不好: "total_surcharges": 5
// Good 好: "surcharge_count": 5
// Good 好: "num_surcharges_applied": 5
```

### Why Case 6 Intermittently Fails: The "total" Trap / 为什么组合编码间歇性失败："total"陷阱

**The Problem / 问题:**
Expected: `"total": 3` (count of items). MiniMax-M2.5 trial 2 returned: `"total": 9.12` (sum of quoted prices: 2.05 + 0.72 + 4.35).

**Root Cause / 根本原因:**

Same ambiguity as Case 4. The field `"total"` in a financial JSON object can mean either:
- Count of records (programmatic convention) / 记录数量（编程惯例）
- Total amount (financial convention) / 总金额（财务惯例）

The model switches interpretation **non-deterministically** between trials — 2 out of 3 trials got it right, but trial 2 computed a financial total instead.

模型在试验之间**非确定性地**切换解释方式——3 次中有 2 次正确，但第 2 次试验计算了财务总额。

## Key Insight: Two Failure Modes / 关键洞察：两种失败模式

All failures across 144 API calls fall into exactly two categories:

所有 144 次 API 调用中的失败恰好分为两类：

### 1. Label Over-Explanation / 标签过度解释
**Affected 影响:** Case 2 (surcharge labels)
**Pattern 模式:** Model correctly identifies the concept but embellishes the output with reasoning context.
**Example 示例:** `"Mirror coating"` → `"Mirror coating (M suffix)"`
**LLM behavior LLM 行为:** Models are trained to be helpful and explanatory. A bare label feels "incomplete" to them, so they add context.

模型正确识别了概念，但在输出中添加了推理上下文来"装饰"结果。模型被训练为有帮助和解释性的。对它们来说，一个裸标签感觉"不完整"，所以它们添加了上下文。

### 2. Count vs. Sum Confusion / 计数与求和混淆
**Affected 影响:** Case 4, Case 6
**Pattern 模式:** Field name like `total` or `total_surcharges` is interpreted as monetary aggregation instead of record count.
**Example 示例:** `"total_surcharges": 5` → `"total_surcharges": 2.88`
**LLM behavior LLM 行为:** In financial documents, "total" overwhelmingly means "sum of money." Models default to this interpretation unless explicitly overridden.

在财务文档中，"total（总计）"绝大多数情况下意味着"金额之和"。模型默认使用这种解释，除非被明确覆盖。

## Recommendations / 建议

### For Skill Authors / 给技能作者

1. **Use unambiguous field names / 使用无歧义字段名**
   - `total` → `item_count` / `项目数量`
   - `total_surcharges` → `surcharge_count` / `附加费数量`
   - `label` → add instruction: "exact text only, no explanations" / 添加指令："仅精确文本，不要解释"

2. **Add output format examples / 添加输出格式示例**
   ```
   Example correct output:
   {"surcharges_detected": [{"flt_code": "8120M", "label": "Mirror coating"}]}
   NOT: {"surcharges_detected": [{"flt_code": "8120M", "label": "Mirror coating (M suffix) +$0.45"}]}
   ```

3. **Separate reasoning from output / 分离推理和输出**
   If models need to explain, give them a separate `reasoning` field so they don't contaminate structured fields.
